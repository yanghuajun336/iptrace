# ADR 0002: 语言与依赖策略采用 Go 主实现 + 最小依赖

- **Status**: Accepted
- **Date**: 2026-03-26

## Context

用户与宪章明确要求：

- 以 Go 为主，C 仅在强耦合且不可替代时才允许；
- 不引入数据库；
- 尽可能减少依赖与代码规模；
- 保障可测试性与可维护性。

研究结论（R-01/R-03）显示：

- 当前需求可用纯 Go 覆盖（规则读取、解析、离线推演、在线追踪）；
- `golang.org/x/sys/unix` 足以支撑 netlink syscall；
- 其余能力可由标准库完成（`flag`、`encoding/json`、`os/exec` 等）。

## Decision

采用以下技术治理策略：

1. **语言策略**：主实现为 Go，当前版本不启用 C/CGo；
2. **依赖策略**：stdlib-first，仅保留 `golang.org/x/sys` 作为外部依赖；
3. **存储策略**：无数据库，规则输入输出使用文件与进程调用；
4. **扩展策略**：新增依赖必须说明收益、维护成本与替代评估。

## Consequences

### Positive

- 构建链路简化，发布与跨环境部署成本低；
- 依赖面更小，安全与升级风险更可控；
- 代码结构更贴近任务与测试拆解，便于持续重构；
- 单二进制构建方式明确，可通过 `go build` 或 `make build` 直接产出交付物。

### Negative

- 某些底层能力需自行封装，前期工程细节会增加；
- 若未来接入更复杂内核特性，可能需要补充依赖评估。

### Follow-up

1. 在版本迭代中定期审视依赖增长；
2. 若未来引入 C/CGo，必须新增 ADR 说明必要性与退场策略；
3. 持续维护基准测试，确保“最小实现”不牺牲性能预算；
4. 保持 `README.md` 与 `Makefile` 中的构建步骤与实际交付方式一致。
